# 卫星任务规划开发的初步计划

我准备开发一个用于敏捷卫星的任务规划算法，由上注模块提供待规划的任务流，而我们准备开发的模块则是：读取该未排序的任务池后将其调度、排序、规划，形成一个任务序列。该任务序列要必须满足硬约束，并努力达到目标函数的利益最大化。

本模块的上流模块是上注模块，它提供待规划的任务池数据；我们负责容器调度等，而我在这部分里负责任务规划组件的开发，注意我们负责的范围。

项目中若有相关的数据缺失，均可以仿真。例如任务池仿真数据，可以是含50-100条任务（含5-10组有向无环图，即依赖链）的任务池，但各字段要完备且仿真效果好，数据不要一眼假。

开发语言采用python，版本3.12.13，python路径：D:/software/anaconda3/python.exe；数值尽量采用配置文件，尽量解耦不要硬编码

本项目预计采用地面基线规划 + 星上自主重规划的模式

## 硬约束

### 1.时间与依赖约束

- 时间窗：任务（执行）开始/结束必须落入可执行窗口。
- 依赖：后继任务开始时刻不早于前置任务完成时刻。数据产品未就绪，不允许触发后处理链路。
- 姿态切换时间开销

### 2.热电约束

受真空散热条件与供电能力上限限制，短时高并发易引发热累积并抬升瞬时功率峰值，进而触发降频、限功率或强制中断等保护动作，严重时还可能导致器件可靠性下降甚至硬件损伤。

### 3.资源与载荷约束

- CPU、GPU、内存、存储、总线、容器并发数不得超配。
- 任务-载荷绑定约束：部分任务可能需要多个载荷支持（载荷类型、载荷数量）；部分任务可能需要某一个特定载荷支持（载荷id、不可替代）

### 4.通信链路约束

- 上注发射、上注接收、遥测回传、数据下行必须落在链路窗口与带宽预算内。

### 5.可靠性与安全约束

- 禁止触发平台定义的危险组合工况（热+功率+姿态冲突）。

## 软约束（尽量满足，作为优化目标）

1. 任务收益最大化（科学价值/业务优先级/时效价值）。
2. 完成率与准时率最大化（降低超时与放弃任务）。
3. 热安全裕度最大化（远离温度上限）。
4. 峰值功率与功率波动最小化（削峰填谷）。
5. 资源利用率最大化（减少空闲，避免无效并发）。
6. 操作平滑性优化（减少频繁启停与姿态频繁切换）。
7. 重编排代价最小化（重规划时的目标）（尽量少改动基线计划，保证可解释性）。

## 算法框架

### 1.地面基线规划

静态全局编排需求，构建融合卫星任务时间窗、温度与瞬时功率等约束的全局任务编排模型。

输入输出：输入无序任务池等数据；输出必须满足硬约束，且尽量靠近优化目标的任务执行基线方案（若是可拆分的最好，便于后续重规划）。（若不合理，请指正）

算法选择：CP-SAT或GA等启发式算法（若不合理，请指正）

其他说明：

- 1.先给出可行域，再在可行域中优化
- 2.给出的方案最好可拆分为子集，便于后续优化以及重规划
- 3.方案可拆分的子集建议：
  3.1.多个滚动规划区间
  PS:滚动规划区间定义：由两个相邻“位置服务”任务的起始时刻界定的时间段；
  3.2或者多个有向无环图（源点为“位置服务”任务或不依赖于其它任务的任务）
- 方案的建模建议：可以包含一组按时间排序的子集、待规划的任务集（为了满足硬约束，可能部分任务无法加入计划中，保留待规划的任务集，便于后续优化）等等
- 关键任务（如“位置服务”任务等）必须被规划

### 2.星上自主重规划

针对在轨运行阶段外部请求突发与热电状态偏差引起的不确定性扰动，构建基于位置感知与热电协同等约束的任务实时重编排机制，该机制对原本的全局容器编排方案进行局部的实时调整，以实现方案的更优化。
构建轻量级闭环流程，识别热电越界风险与潜在机会。
应用场景例如，芯片实时温度过高，需要调整后续的任务流，以实现降温；部分任务由于特殊情况，到预定时间并未启动，导致计算资源闲置，该机制需调整任务序列以实现计算资源的合理利用。

输入输出：输入任务执行基线方案、卫星实时状态等数据；输出：重规划后的任务执行方案（局部重规划，与原来的方案拼接）（若不合理，请指正）

创新点与难点：准备采取强化学习的方案，能根据当前卫星实时状态（轨道位置，热电物理状态等）、全局基线任务序列等来给出任务实时重编排方案。但目前相关数据集是缺失的，我该如何解决？比如前期采用相关启发式规则来调整任务序列，当相关软件运行几个周期后，有一定的参考数据了，引入强化学习相关机制，通过强化学习和启发式规则相结合的方式来调整任务序列

其他说明：

- 重规划范围不采用“剩余全部时间”默认口径，采用分级有限范围：

  - 基础范围：当前滚动重规划区间的剩余部分。
  - 扩展范围：当前滚动重规划区间 + 下一个滚动重规划区间。
  - 严重范围：再按上限扩展到后续若干区间（配置化上限）。
- 触发机制：避免过多的重规划，重规划应该有触发机制。ps：重规划响应时限指从“出现闲置/危险状态”到“触发重规划事件”的时长。重规划求解耗时不并入“响应时限”。

  - 触发分级遵循严重度优先：L3 立即，L2 次之，L1 最慢
  - 情况分类：L3 （紧急）（紧急注入、热电危险等情况），L2（告警），L1 （可优化）
  - 对于待规划、未规划的任务池的触发机制：任务累计价值超过阈值
  - 最小收益门槛保护：对非紧急触发增加门槛，若 `predicted_replan_gain < min_replan_gain`，可延后触发并并入后续重规划。
