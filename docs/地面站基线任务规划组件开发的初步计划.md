# 卫星任务规划开发的初步计划

我准备开发一个用于敏捷卫星的任务规划算法，由上注模块提供待规划的任务流，而我们准备开发的模块则是：读取该未排序的任务池后将其调度、排序、规划，形成一个任务序列。该任务序列要必须满足硬约束，并努力达到目标函数的利益最大化。

本模块的上流模块是上注模块，它提供待规划的任务池数据；我们负责容器调度等，而我在这部分里负责任务规划组件的开发，注意我们负责的范围。

项目中若有相关的数据缺失，均可以仿真。目前输入数据为data\latest_small_tasks_pool.json，data\latest_windows.json分别是一组任务池与时间窗数据示例

开发语言采用python，版本3.12.13，python路径：D:/software/anaconda3/python.exe；数值尽量采用配置文件，尽量解耦不要硬编码

现阶段我们先完成地面基线规划（现阶段重点），后续再加上星上自主重规划（预计前期规则启发后期引入强化学习，不是现阶段目标）

## 硬约束

### 1.时间与依赖约束

- 时间窗：任务（执行）开始/结束必须落入对应窗口。
- 依赖：后继任务开始时刻不早于前置任务完成时刻。
- 姿态切换时间开销（卫星此时的姿态不是任务要求的目标姿态，则需执行姿态切换任务或者等待一段时间完成姿态切换）

### 2.热电约束

受真空散热条件与供电能力上限限制，短时高并发易引发热累积并抬升瞬时功率峰值，进而触发降频、限功率或强制中断等保护动作，严重时还可能导致器件可靠性下降甚至硬件损伤。（暂定热累计由任务功率与实时温度计算出）

### 3.资源与载荷约束

- CPU并发数、GPU、内存等不得超配。
- 任务-载荷绑定约束：部分任务可能需要多个载荷支持（载荷类型、载荷数量）；部分任务可能需要某一个特定载荷支持（载荷id、不可替代）

## 软约束（尽量满足，作为优化目标）

1. 任务收益最大化（科学价值/业务优先级/时效价值）。
2. 完成率与准时率最大化（降低超时与放弃任务）。
3. 热安全裕度最大化（远离温度上限）。
4. 峰值功率与功率波动最小化（削峰填谷）。
5. 资源利用率最大化（减少空闲，避免无效并发）。
6. 操作平滑性优化（减少频繁启停与姿态频繁切换）。
7. 重编排代价最小化（重规划时的目标）（尽量少改动基线计划，保证可解释性）。

## 算法框架

### 1.地面基线规划

静态全局编排需求，构建融合卫星任务时间窗、温度与瞬时功率等约束的全局任务编排模型。

输入输出：输入无序任务池等数据；输出必须满足硬约束，且尽量靠近优化目标的任务执行基线方案（解集是由子集构成的最好，便于后续重规划）。（若不合理，请指正）

算法选择：CP-SAT或者遗传算法/粒子群等启发式算法（选择较优的方案；若不合理，请指正）

其他说明：

- 先给出绝对不违背硬约束的可行解域，再在可行解域中优化
- 给出的解方案可拆分为子解（若不好实现则放在后期目标），便于后续优化以及重规划
- 解方案的建模建议：可以包含一组按时间排序的任务集集、待规划的任务集（为了满足硬约束，可能部分任务无法加入计划中，保留待规划的任务集，便于后续优化）等等
- 给出的解能体现并行性
- 有事件驱动或状态记录组件，能记录每个时刻载荷、资源使用/空闲情况，热电状态，卫星仰角等（保证不超限），能考虑资源如内存的生命周期（比如group01_radar任务的内存占用，可能得等后续group01_storage任务边执行边释放）
- 由求解器的迭代日志记录组件（如果CP-SAT是迭代的，每隔10或100代记录下这代的解）
