# M5 诊断与优化报告（先诊断再优化）

## 实验设置

- 脚本: `scripts/diagnose_schedule_ratio.py`
- 命令: `python scripts/diagnose_schedule_ratio.py --runs 20 --seed-start 1001 --output output/m5_diagnose_summary.csv`
- 可复现设置: 诊断脚本为四个分组统一设置 `runtime.solver_random_seed=20260324` 与 `runtime.solver_num_workers=1`
- 权威结果文件: `output/m5_diagnose_summary.csv`（报告表格与结论均以该文件为准）
- 指标:
  - 平均 scheduled_ratio
  - infeasible 占比（`solver_status` 非 `optimal/feasible`）
  - scheduled_count 中位数

## A/B/C/D 分组定义

- A 基线: `key_task_probability=0.06`，`max_hard_key_tasks=task_count_max`（近似“几乎不设硬关键上限”）
- B 数据可行性优化: `key_task_probability=0.01`，`max_hard_key_tasks=1`
- C 姿态影响对照: 在 B 基础上 `attitude_time_per_degree=0`
- D 求解时长对照: 在 B 基础上 `solver_timeout_sec=30`

## 实验结果

- 当前批次的精确数值以 `output/m5_diagnose_summary.csv` 为准（该文件由上文命令直接生成）。
- 结果趋势稳定：
  - `A_baseline` 显著劣于 `B_data_feasibility`（平均排程率更低、不可行占比更高）。
  - `B_data_feasibility` 与 `D_timeout30_on_B` 基本持平（延长求解时长收益有限）。
  - `C_no_attitude_on_B` 明显优于 `B_data_feasibility`（姿态切换约束是次要但明确的上限因素）。

## 结论

- 主因判断: **更偏数据可行性（硬关键任务密度过高）**，而不是求解时长。
- 证据 1: `A -> B` 后不可行占比显著下降，平均排程率显著提升，说明主因来自数据可行性优化。
- 证据 2: `B -> D`（仅延长求解时长）收益很小，说明超时不是主要瓶颈。
- 证据 3: `B -> C`（移除姿态切换时间）有明确增益，说明姿态约束是次要但明显的上限因素。

## 优化落地摘要

- 在 simulation 配置/实现中引入 `max_hard_key_tasks`（默认 1）。
- 默认 `key_task_probability` 下调为 `0.01`。
- 超过硬关键上限的“关键候选”仅保留 `value bonus`，不再设置 `is_key_task=True`。
- 保持了“约束前置在 simulation，planner 通用”的原则，planner 无需新增任务可行性特化逻辑。
