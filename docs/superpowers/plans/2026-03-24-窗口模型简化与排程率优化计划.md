# 窗口模型简化与排程率优化计划（/writing-plans）

> 适用范围：.worktrees/ai-develop 分支上的下一轮重构。
> 
> 目标对应你提出的 5 点：
> 1. 移除 VisibilityWindow.window_type；
> 2. 持久化可见窗口数据；
> 3. 简化 simulation.json 冗余概念；
> 4. 下调 flex（自由任务）占比；
> 5. 对 scheduled_count 偏低做根因分析并优化。

## 一、现状与根因假设

### 1) 现状证据（已观察）
- 当前 `free_task_ratio=0.35`，自由任务比例偏高。
- 诊断脚本基线结果显示平均排程率约 `0.159`，且多组 seed 出现 `solver_status=3` 且 `scheduled=0`。
- `solver_status=3` 在 OR-Tools CP-SAT 中对应不可行（infeasible）。

### 2) 根因假设（优先级）
- 高优先级根因A：硬性关键任务过多（`key_task_probability` 使随机任务也变为 `is_key_task=True`），与窗口、依赖、姿态切换共同作用后导致模型不可行。
- 高优先级根因B：窗口类型绑定（`window_type`）导致任务可分配窗口集合过窄，放大不可行概率。
- 中优先级根因C：序列/DAG/共享窗口三套参数叠加，导致生成数据结构过于“硬”，可行域缩小。
- 中优先级根因D：自由任务比例偏高稀释了可在窗口内高价值完成的任务密度，影响 scheduled_count 与 objective。

## 二、总体改造原则

- 任务硬约束优先前置到 simulation 生成阶段，planner 保持通用求解器角色。
- 先做“语义降复杂”（去类型、去冗余参数），再做“可行性提纯”（降低硬关键任务密度、控制依赖密度），最后做“指标回归”。
- 每一步均配套测试与可观测产物，防止回归。

## M3：简化 simulation.json 冗余概念

### 目标
把 `sequence*`、`dag_chains*`、`window_share_task*` 三组重叠参数收敛成更直接的控制面。

### 新参数草案（建议）
- `structured_task_ratio`: 窗口约束任务占比（其余为自由任务）。
- `dependency_density`: 结构化任务中带前驱的概率。
- `window_reuse_target`: 单窗口期望绑定任务数（均值目标）。
- `window_count_min/max`、`window_duration_min/max`（保留）。
- `free_task_ratio` 保留但下调，或直接由 `1 - structured_task_ratio` 推导（二选一，优先后者）。

### 变更范围
- `config/simulation.json`
- `src/scheduler/config.py`
- `src/scheduler/simulation.py`
- `tests/test_config.py`
- `tests/test_simulation.py`
- `docs/运行说明.md`

### 实施步骤
- [ ] 删除冗余旧键并添加兼容迁移（旧键存在时给 warning，后续版本移除）。
- [ ] 用新参数重写生成流程，避免“序列长度 + DAG链数 + 窗口共享阈值”三套并行。
- [ ] 更新文档与示例配置。

### 验收标准
- 新配置可独立驱动仿真，不依赖旧键。
- 旧配置有清晰告警且行为可预测。

## M4：降低 flex 占比并提高可调性

### 变更范围

- `tests/test_simulation.py`

### 实施步骤

- [x] 增加下限保护：结构化任务至少占 60%。

### 验收标准
- 多 seed 统计下，自由任务占比稳定在目标区间。
- 任务池结构更接近“窗口任务为主，自由任务补充”。

## M5：scheduled_count 低问题的根因定位与优化

### 诊断实验设计
- 对照组A（基线）：当前参数。
- 对照组B：仅降低 `key_task_probability`（如 `0.06 -> 0.01`，并限制硬关键任务最大数量，如 `max_hard_key_tasks=1`）。
- 对照组C：保持B，再关闭姿态切换时间（`attitude_time_per_degree=0`）用于隔离姿态影响。
- 对照组D：保持B，再提升求解时限（如 `solver_timeout_sec=30`）观察是否为搜索不足。

### 优化动作（按优先级）
- [ ] 引入“硬关键任务上限”机制：仅少量任务 `is_key_task=True`，其他改为高价值软优先。
- [ ] 在 simulation 中做前置可行性剪枝：若窗口内无法容纳必选依赖链则降级为软关键或改绑窗口。
- [ ] 放宽窗口绑定策略（去类型后自动生效），提升可行域。
- [ ] 若 B 显著改善而 D 改善有限，说明主因在数据可行性而非求解时长，优先固化数据侧修复。

### 指标门槛（建议）
- 多 seed（>=20）平均 scheduled_ratio 提升到 `>=0.45`。
- `solver_status=3` 占比降到 `<10%`。
- `scheduled_count` 中位数较当前基线提升至少 50%。

## 四、测试与回归清单

- [ ] 单元测试：模型字段、仿真分布、配置校验、pipeline 输出契约。
- [ ] 端到端：`python -m pytest -q` 全量通过。
- [ ] 冒烟运行：`python main.py --config-dir config --seed 42`。
- [ ] 产物核验：`latest_task_pool.json` 与 `latest_visibility_windows.json` 字段一致性。
- [ ] 指标脚本：输出各 seed 的 `tasks/scheduled/status`，并生成均值、中位数、不可行占比。

## 五、执行顺序建议

1. 先做 M1 + M2（语义与可观测性打底）。
2. 再做 M3 + M4（参数与分布简化）。
3. 最后做 M5（诊断-调参-固化），并以量化指标验收。

## 六、风险与回退

- 风险1：去除 `window_type` 后窗口过于“宽松”，可能弱化场景真实性。
  - 对策：通过 payload 资源容量和依赖密度维持区分度，不再用窗口类型硬编码。
- 风险2：降低硬关键任务后，关键业务任务覆盖下降。
  - 对策：区分“硬关键（must schedule）”与“软关键（high value）”两级优先级。
- 风险3：配置迁移期间旧键与新键并存引发歧义。
  - 对策：定义优先级与告警规则，并在文档给出迁移映射表。
