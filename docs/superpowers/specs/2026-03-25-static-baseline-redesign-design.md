# 静态基线规划重建设计（一期）

日期：2026-03-25  
分支：feature/ai-develop  
范围：仅基于 ai-develop 当前上下文，从头重建静态基线规划能力；除模型与给定静态样例数据外，不复用历史实现逻辑。

## 1. 目标与范围

### 1.1 一期目标

1. 输入固定为静态任务池与可见窗口数据，结合领域模型完成规划。
2. 输出满足硬约束的可执行调度结果。
3. 关键任务采用“尽量必排”策略：优先排入，不可排时必须输出结构化原因。
4. 使用混合策略：启发式快速构造初解，CP-SAT 在限时内改进。

### 1.2 一期边界

1. 仅实现地面静态基线规划，不实现在线重规划决策流程。
2. 允许预留重规划接口（数据结构与入口），但不实现重规划求解。
3. 主入口和核心流程只依赖新架构，不回退到旧 main 分支历史逻辑。

### 1.3 一期成功标准

1. 产出稳定：已排任务序列、未排任务与原因、关键指标摘要完整。
2. 可复现：同输入与同随机种子下结果一致（允许定义等价顺序比较规则）。
3. 可退化：在给定超时内始终返回结果（可行解或失败诊断）。
4. 迭代可观测：若算法处于迭代阶段，至少每 10 代输出一次中间解摘要。
5. 日志契约：迭代日志文件在求解前创建；提前结束时仍写入终态摘要。

## 2. 方案比较与选型

### 2.1 备选方案

1. 纯 CP-SAT：最优性潜力高，但重建周期与调参成本较大。
2. 纯启发式：实现快、稳定，但质量上限受限。
3. 混合策略：启发式初解 + CP-SAT 限时改进。

### 2.2 选型结论

采用方案 3（混合策略）。

理由：

1. 兼顾交付效率与解质量。
2. 先有可行再做优化，降低无解卡死风险。
3. 与“高效但保证质量”的目标一致。
4. 为后续重规划保留“快速可行 + 局部改进”路径。

## 3. 架构与模块边界

### 3.1 输入层（Input Adapter）

职责：

1. 读取静态任务与窗口数据。
2. 完成字段级校验与基础归一化。
3. 输出标准化任务集合与窗口索引。

非职责：

1. 不做求解策略选择。
2. 不做复杂优化。

### 3.2 预处理层（Problem Builder）

职责：

1. 构建依赖图与拓扑顺序。
2. 构建时间域与窗口映射。
3. 构建姿态切换代价矩阵。
4. 构建统一问题对象（供启发式与 CP-SAT 共用）。

### 3.3 初解层（Heuristic Scheduler）

职责：

1. 基于关键任务优先、可行性优先、收益密度生成初解。
2. 输出首个可行计划与未排任务原因。

### 3.4 改进层（CP-SAT Improver）

职责：

1. 接收初解作为 warm start。
2. 在时限内迭代改进目标值。
3. 每 10 代记录中间摘要，结束写终态摘要。

### 3.5 输出层（Result Writer）

职责：

1. 序列化计划、未排原因、指标与求解摘要。
2. 输出迭代日志（jsonl）。
3. 保证 schema 稳定与可诊断性。

### 3.6 重规划接口（Reserved API）

职责：

1. 仅定义输入/输出结构与调用入口。
2. 一期不实现重规划算法主体。

## 4. 数据流与约束落位

### 4.1 端到端数据流

1. 输入解析：读取并校验任务与窗口。
2. 预处理建模：生成依赖图、时间域、代价矩阵与资源向量。
3. 启发式初排：快速生成可行解。
4. CP-SAT 改进：限时优化并输出迭代摘要。
5. 结果汇总：输出计划、诊断与指标。

### 4.2 约束分层原则

放在数据与预处理层：

1. 窗口引用合法性。
2. 依赖引用完整性与无环性。
3. 字段合法域约束。

放在求解层：

1. 时间窗约束。
2. 依赖先后约束。
3. 资源容量约束（CPU/GPU/内存/功率）。
4. 姿态切换时间开销约束。

规则：

1. 同一硬约束不重复编码于多层。
2. 任务可行性尽量前置，求解器保持通用优化职责。

### 4.3 目标函数分层

1. 第一层：最大化总收益，优先保障关键任务完成。
2. 第二层：最小化关键任务未排惩罚、超窗惩罚、姿态频繁切换惩罚。
3. 第三层：优化资源平滑与功率波动。

### 4.4 未排任务原因标准化

主标签枚举：

1. window_infeasible
2. dependency_blocked
3. resource_conflict
4. attitude_transition_insufficient
5. solver_timeout_best_effort

每个未排任务至少输出一个主标签与可读说明。

## 5. 错误处理与可观测性

### 5.1 错误分层

1. 输入错误：立即失败并返回明确错误。
2. 规划错误：进入 best-effort 输出并给出诊断。
3. 系统错误：中断并保留诊断上下文。

### 5.2 输出与日志契约

主结果文件包含：

1. schedule
2. unscheduled
3. metrics
4. solver_summary

迭代日志（jsonl）规则：

1. 求解前先创建文件。
2. 默认每 10 代写一条摘要。
3. 至少一条终态摘要（completed/timeout/infeasible）。

单条迭代摘要建议字段：

1. iteration
2. timestamp
3. objective_value
4. scheduled_task_count
5. key_task_scheduled_count
6. unscheduled_task_count
7. feasible
8. improvement_delta
9. note

### 5.3 核心观测指标

1. 结果质量：total_value、key_task_completion_ratio、scheduled_task_ratio。
2. 过程质量：first_feasible_iteration、best_solution_iteration、iteration_count、no_improvement_streak。
3. 运行效率：total_runtime_ms、heuristic_runtime_ms、cpsat_runtime_ms。

## 6. 测试策略与验收

### 6.1 测试分层

1. 单元测试：解析校验、依赖图、姿态代价、原因分类。
2. 组件测试：启发式可行性、CP-SAT 限时改进、关键任务策略。
3. 集成测试：全链路、可复现、输出 schema 与日志契约。

### 6.2 一期最小验收用例

1. 基线可行性：样例输入可产出合法调度。
2. 关键任务不可排场景：系统返回结构化原因。
3. 迭代可观测性：每 10 代摘要 + 终态摘要 + 先建日志文件。
4. 超时韧性：短超时下返回 best-effort 与诊断。
5. 可复现性：固定 seed 下关键指标一致。

## 7. 里程碑划分

1. M1：新架构骨架与主入口切换。
2. M2：静态输入与预处理实现。
3. M3：启发式初解实现。
4. M4：CP-SAT 限时改进与迭代日志实现。
5. M5：统一输出、测试与验收。

## 8. 风险与应对

1. 风险：模型复杂导致 CP-SAT 收敛慢。  
   应对：保底启发式可行解 + 严格超时 + 分层目标。

2. 风险：日志有开关但无产物。  
   应对：日志文件前置创建与终态兜底写入。

3. 风险：从头重建过程中旧代码回流。  
   应对：主入口仅依赖新模块，旧模块不纳入新链路。

## 9. 实施前冻结约定

1. 一期严格限定为静态基线，不扩展在线重规划实现。
2. 若需求新增，先更新设计文档再进入实现。
3. 实现阶段遵循测试先行与里程碑式提交。
